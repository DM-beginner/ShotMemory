# 文档编写准则

**适用范围**：本仓库所有 `.md` 文档（`README.md`、`CLAUDE.md`、`AGENTS.md`、`docs/**`）的撰写与更新——人写也好，AI 协作也好，都按这份走。

**目的**：避免文档腐烂（rot）。腐烂的文档比没有文档更糟——它会误导读者去做错事。

---

## 1. 四条核心原则

任何文档改动都必须能用其中至少一条解释"为什么这么写"。

1. **代码能说清楚的，文档不再说**。表结构看 `models/`，迁移看 `alembic/versions/`，路由签名看 `routers/`，接口契约看 `/v1/docs`（OpenAPI 自动生成）。
2. **跨文件引用只到符号，不到行号**。`auth_router._set_auth_cookies` 抗腐烂；`auth_router.py:39` 一次重构就过时。
3. **先答 Why 再答 What**。文档的核心价值是解释"为什么这么设计"、"什么必须保持"、"什么不能这么做"。What（实现细节）让代码去说。
4. **单一信源**。每个事实只在一个地方写，其他地方需要时只链接，不抄写。

---

## 2. 该写进文档的（KEEP）

| 类别 | 例子 |
|------|------|
| 意图与非意图 | "本项目不做多用户共享照片墙" |
| 不变量（Invariants） | "Worker 必让 `exif_data` 从 null 翻成非 null，不会停在 processing" |
| 约束（Constraints） | "上传单批 ≤ 50 张、单文件 ≤ 50 MB、All-or-Nothing 校验" |
| 决策与权衡 | "为什么选 arq 而非 Celery"——含主要替代方案，放 `docs/decisions.md` |
| 跨边界契约 | "`object_key` 派生路径规则"、"`status` 状态机契约"、"`PhotoResponse.thumbnail_key` 条件返回语义" |
| 领域知识 | EXIF 命名空间优先级 / GPS (0,0) 过滤 / H.264 vs H.265 转码策略 |
| 非显然的业务规则 | "软删除账号立即释放邮箱供他人复用，靠部分唯一索引" |
| 运维与故障应对 | 队列卡住怎么排查、Redis 挂了主服务什么行为——放 `docs/runbook.md` |
| 已知未做项 | 显式列出"占位 / 未实现"——集中在 `docs/production-readiness.md` |
| 项目特有术语 | `device_id` / `ContentIdentifier` / `pre_video_key`——放 `docs/glossary.md` |

---

## 3. 不该写进文档的（CUT）

| 反模式 | 替代做法 |
|--------|---------|
| 完整 DB 字段表（每列类型 + 说明） | 只列非显然字段；类型以 ORM 模型为准 |
| API 端点 Request/Response 全 JSON 样例 | 端点清单 + 非显然请求语义；完整字段去 `/v1/docs` |
| 函数签名 / 代码贴片 / try-except 块 | 描述行为与失败语义，不贴实现 |
| `file.py:NN` 行号引用 | 改用符号名（函数 / 类 / 常量） |
| 项目目录树 | 不写。读者 `ls` / 用 IDE 即可 |
| 营销文案（"亮点"、"特性"、"全链路异步"等） | 删。文档不是 README badge |
| 实施进度 / 完成百分比 | 放 issue tracker，不进文档 |
| 历史叙事（"我们最初用 X，后来换 Y"）| 除非能解释"现在为什么是 Y"，否则删。git log 已有 |
| "TODO / FIXME / 占位" 散落在十几个文件 | 集中收敛到 `docs/production-readiness.md` |
| 重复（README、CLAUDE.md、architecture.md 抄同一段） | 单一信源，其他地方只链接 |

---

## 4. 格式规范

- **链接**：用相对路径，链到文件或目录级，**不要链 Markdown 锚点**——锚点会随标题改名而失效。  
  ✅ `[`docs/modules/auth.md`](../modules/auth.md)`  
  ❌ `[设备语义](./auth.md#6-设备语义)`
- **表格**：列数 ≤ 5；长说明在表外补段落，不要把表格撑成多行长文本。
- **代码块**：只贴 schema、接口契约、状态机、配置项；不贴业务实现。
- **ASCII 图**：保留——状态机、跨进程时序、关系拓扑这些图用 ASCII 比文字快。
- **emoji**：除非用户明确要求，不加。
- **段落顶部**：每个文档第 1 段必须能让陌生读者在 30 秒内回答"这文档讲什么、给谁看"。

---

## 5. 文档归属表

每类信息只写在唯一信源里，其他位置只放指针。

| 内容 | 唯一信源 |
|------|---------|
| 一句话项目定位 | `README.md` |
| 环境要求 + 快速开始命令 | `README.md` |
| 开发命令（全套 lint / test / migration） | `CLAUDE.md` |
| AI 协作守则 | `CLAUDE.md` |
| 完整后端架构 | `docs/architecture.md` |
| 前端架构 | `docs/frontend-architecture.md` |
| 模块业务规则、API 接口语义、数据模型 | `docs/modules/<模块>.md` |
| 跨模块流程（含时序图） | `docs/flows/<流程>.md` |
| 关键设计决策（ADR）| `docs/decisions.md` |
| 项目术语 | `docs/glossary.md` |
| 故障与运维 | `docs/runbook.md` |
| 已知未做 / 占位实现 | `docs/production-readiness.md` |

---

## 6. 维护守则（什么时候要同步改文档）

代码改动时，对应文档段同步更新：

| 代码改动 | 需同步的文档段 |
|---------|---------------|
| 改 schema / 加迁移 | 对应模块 `.md` 的"数据模型"段（仅当涉及非显然字段或约束） |
| 加 / 改路由 | 对应模块 `.md` 的"API 接口"段（端点清单 + 非显然请求语义） |
| 改业务规则 / 校验 | 对应模块 `.md` 的"业务规则"段 |
| 加新模块 | `CLAUDE.md` 的 "Detailed Documentation" 索引 + 新建 `docs/modules/<新>.md` |
| 新增 / 移除依赖关系 | 对应模块 `.md` 的"依赖关系"段 |
| 引入新的横切设计决策 | `docs/decisions.md` 加一条 ADR |
| 引入项目特有新术语 | `docs/glossary.md` 加一行 |
| 加运维操作 / 故障路径 | `docs/runbook.md` |
| 标记某项为"占位 / 待实现" | `docs/production-readiness.md` |

**反向规则**：如果代码改动后这些段都不需要改，那这次改动多半是纯实现细节——文档不动是正确的。**不要为了"看起来在维护文档"而强行改它**。

---

## 7. 自检清单

每次 `.md` 文件改动完成后，跑一遍：

```bash
# 1. 残留行号引用应为 0
grep -rn "\.\(py\|ts\|tsx\):[0-9]" docs/ README.md CLAUDE.md
```

- [ ] 该文件第 1 段能在 30 秒内回答"给谁看、看完拿到什么"
- [ ] 没有复述 ORM / schema / 路由签名能直接读出来的内容
- [ ] 没有贴出 `try / except`、for 循环、async with 之类的实现代码
- [ ] 新增"占位 / 未实现"项已同步进 `docs/production-readiness.md`
- [ ] 新增术语已同步进 `docs/glossary.md`
- [ ] 新增设计决策已同步进 `docs/decisions.md`
- [ ] 这段信息在仓库内**只在这一处**（搜不到第二个副本）
- [ ] 链接都是文件 / 目录级，没有 `#锚点`

## 8. AI 协作时的额外约束

AI 在更新文档时容易踩的坑：

1. **不要在描述代码细节时凭空编造**：如果不确定某个函数行为，去读源码再写，不要根据函数名猜。
2. **不要补"假装存在"的内容**：例如不要写"已接入 Sentry 告警"——除非真接入了。已知未做的事写"未做"，比写"假装做了"诚实十倍。
3. **不要把"删除"做成"先注释"**：注释保留的旧内容会引诱后来者以为是"暂时禁用"——直接删，需要时 `git log` 找回。
4. **不要把不相关改动一起塞进来**：文档更新就只改文档，不要顺手"优化"代码逻辑。一次 PR 一个意图。
5. **不要主动加 Goals/Non-Goals**：除非用户明确要求。个人项目演化方向不固定，写死反而成束缚。
6. **不要主动改 README 加"项目亮点"**：营销内容只在用户明确说"这是要给外人看的"时才加。
7. **改完后必须自跑第 7 节的清单**，把残留行号、占位散落等问题清掉再交付。

---

## 9. 一句话总结

> 文档写"代码说不出来的"，删"代码已经说清楚的"，绝不写"和代码冲突的"。
